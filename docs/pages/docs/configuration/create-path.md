`createPath` is a small helper exported from `zudoku` that lets you define an internal route path
**once** and reference it everywhere it's needed.

```ts
import { createPath } from "zudoku";

const apiReference = createPath("/api");
```

## Why it exists

Pages generated by a plugin (such as an [API reference](./api-reference.md) or an
[API catalog](./api-catalog.md)) are configured in two separate places:

1. The plugin config, where you set the `path` the plugin is **mounted** at.
2. The [navigation](./navigation.mdx), where a `link` item's `to` points at that same path so users
   can reach it.

It's easy to miss that these two values have to match, and just as easy to change one without
changing the other. When they drift apart, the navigation link points at a path where nothing is
mounted — a broken link with no obvious cause.

`createPath` solves this by giving you a single value to reference in both places:

```ts title=zudoku.config.ts
import { createPath, type ZudokuConfig } from "zudoku";

// Defined once...
const apiReference = createPath("/api");

const config: ZudokuConfig = {
  navigation: [
    // ...referenced in the navigation...
    { type: "link", to: apiReference, label: "API Reference" },
  ],
  apis: {
    type: "file",
    input: "./openapi.json",
    // ...and referenced again where the plugin is mounted.
    path: apiReference,
  },
};

export default config;
```

Because both the link and the mount point read from the same `apiReference` value, they can never
drift out of sync.

## How it works

`createPath` returns the path string unchanged — it only validates that the path is non-empty. The
value can be an absolute path (`"/api"`) or a relative segment (`"api-users"`) that you compose with
[`joinUrl`](#sub-paths). The returned value is a `PathReference`, a string under the hood, so you
can use it **anywhere a path string is expected**:

```ts title=zudoku.config.ts
import { createPath, joinUrl, type ZudokuConfig } from "zudoku";

const apiReference = createPath("/api");

const config: ZudokuConfig = {
  navigation: [{ type: "link", to: apiReference, label: "API Reference" }],
  apis: { type: "file", input: "./openapi.json", path: apiReference },
  // Works in redirects...
  redirects: [{ from: "/", to: apiReference }],
  // ...and anywhere else a path is accepted, e.g. protected routes.
  protectedRoutes: [joinUrl(apiReference, "*")],
};

export default config;
```

:::note

`createPath` does not normalize the path. A value like `createPath("/api/")` is returned as-is.
Whatever value you pass is used consistently in every place that references it.

:::

## Catching duplicate paths

`createPath` keeps an internal registry of the **absolute** paths created while your config is
evaluated. Creating the same absolute path twice throws an error, because two things can't be
mounted at the same route:

```ts
const apiReference = createPath("/api");
const duplicate = createPath("/api"); // ❌ throws: createPath("/api") was called more than once
```

The fix is the whole point of `createPath`: create each path **once**, assign it to a variable, and
reference that variable everywhere it's needed.

Only absolute paths are checked. Relative segments — the plain strings you pass to `joinUrl` to
build a deeper path — are not registered, since the same segment (e.g. a version like `"v1"`) can
legitimately appear under several different bases.

## Examples

### Multiple APIs

Define a path for each API and reference them from both the `apis` array and the navigation:

```ts title=zudoku.config.ts
import { createPath } from "zudoku";

const usersApi = createPath("/apis/users");
const billingApi = createPath("/apis/billing");

const apis = [
  { type: "file", input: "./users.json", path: usersApi },
  { type: "file", input: "./billing.json", path: billingApi },
] as const;

const navigation = [
  { type: "link", label: "Users API", to: usersApi },
  { type: "link", label: "Billing API", to: billingApi },
] as const;
```

See [Multiple APIs](../guides/using-multiple-apis.md) for the full configuration.

### Sub-paths

To build a deeper path from a base — for example a nested API mount or a link to a specific version
— compose it with the `joinUrl` helper (also exported from `zudoku`), wrapping the new segment in
`createPath`:

```ts title=zudoku.config.ts
import { createPath, joinUrl } from "zudoku";

const shipmentsApi = createPath("/api-shipments");

const config = {
  navigation: [
    // Links to a page nested under the API
    {
      type: "link",
      label: "Track a shipment",
      to: joinUrl(shipmentsApi, "track-shipment"),
    },
  ],
  apis: { type: "file", input: "./shipments.json", path: shipmentsApi },
};
```

`joinUrl` handles the slashes between segments, so `joinUrl(shipmentsApi, "track-shipment")`
resolves to `"/api-shipments/track-shipment"`. Prefer `joinUrl` over string interpolation so the
base path stays the single source of truth.

### API Catalog

When using an [API Catalog](./api-catalog.md), define the catalog path once and derive each API's
path from it so the nested paths stay consistent with the catalog:

```ts title=zudoku.config.ts
import { createPath, joinUrl } from "zudoku";

const catalog = createPath("/catalog");

const config = {
  catalogs: { path: catalog, label: "API Catalog" },
  navigation: [{ type: "link", label: "API Catalog", to: catalog, icon: "square-library" }],
  apis: [
    {
      type: "file",
      input: "./users.json",
      path: joinUrl(catalog, "api-users"),
      categories: [{ label: "General", tags: ["Users"] }],
    },
  ],
};
```
