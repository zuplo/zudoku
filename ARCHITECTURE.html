01System overview

Zudoku is a Vite-orchestrated React app. A user writes a zudoku.config.tsx, drops in MDX and OpenAPI schemas, and the framework assembles a routable, themed, plugin-extensible documentation site that can be served live or prerendered to static HTML.

04Build pipeline · the Vite plugin

One vitePlugin() returns an array of 17 sub-plugins. Each adds capability via virtual modules that the runtime imports. This is where most of the magic happens at build time. 

// packages/zudoku/src/vite/plugin.ts
export default function vitePlugin(): PluginOption {
  return [
    viteShikiPlugin(),           // syntax highlighting bundle
    viteConfigReloadPlugin(),    // HMR on zudoku.config.tsx
    viteMdxPlugin(),             // MDX → React + custom remark/rehype
    react({ include: /\.(mdx?|jsx?|tsx?)$/ }),
    viteConfigPlugin(),          // virtual:zudoku-config
    viteApiKeysPlugin(),         // virtual:zudoku-api-keys-plugin
    viteCustomPagesPlugin(),     // virtual:zudoku-custom-pages-plugin
    viteAuthPlugin(),            // virtual:zudoku-auth
    viteDocsPlugin(),            // MDX docs route registration
    viteDocMetadataPlugin(),     // frontmatter, last-modified
    viteNavigationPlugin(),      // virtual:zudoku-navigation
    viteApiPlugin(),             // SchemaManager: OpenAPI codegen
    viteSearchPlugin(),          // pagefind / inkeep / algolia wiring
    viteMarkdownExportPlugin(),  // /llms.txt + raw .md exports
    vitePluginSsrCss({ entries: ["zudoku/app/entry.server.tsx"] }),
    viteThemePlugin(),           // shadcn theme registry, CSS vars
    tailwindcss(),               // Tailwind v4 (CSS-first)
  ];
}

05The plugin contract

A ZudokuPlugin is a discriminated union — any object satisfying any of the shapes below qualifies. Plugins are duck-typed at runtime via isXxxPlugin() guards. 

// packages/zudoku/src/lib/core/plugins.ts
export type ZudokuPlugin =
  | CommonPlugin           // initialize · getHead · getMdxComponents
  | NavigationPlugin       // getRoutes · getNavigation · getProtectedRoutes
  | ApiIdentityPlugin      // getIdentities (for playground auth)
  | SearchProviderPlugin   // renderSearch
  | ProfileMenuPlugin      // getProfileMenuItems
  | EventConsumerPlugin    // events: { onNavigate, onAuthChange, ... }
  | AuthenticationPlugin   // signIn, signUp, signOut, getAccessToken
  | TransformConfigPlugin; // transformConfig (build-time mutation)
Build-time hook

TransformConfigPlugin

Mutates the resolved ZudokuConfig before runtime. Used for the with-zuplo integration, processor injection, etc.

Runtime hook

NavigationPlugin

The most powerful kind. Owns routes, can expose a custom navigation tree, and can gate routes behind auth.

Runtime hook

CommonPlugin

The lightest contract. getHead() injects into <head> per location, getMdxComponents() adds custom MDX tags.

07OpenAPI · the two-pipeline story

The OpenAPI plugin runs two distinct processing pipelines depending on where the schema comes from. Both end at the same place: a GraphQL Pothos schema queried in the browser by React components.

Build-time File schemas · SchemaManager

Runs once in the Vite build via vite/api/SchemaManager.ts. The result is dumped to node_modules/.zudoku/processed and codegenned into a static import.

Step 1
$RefParser.bundle()
Bundles external $refs, keeps internal refs. preservedProperties: ["description", "summary"].
Step 2
@scalar/openapi-parser upgrade()
3.0 → 3.1. example → examples. Skipped if already 3.1+.
Step 3
flattenAllOfProcessor
Resolves $refs inside allOf arrays then merges via @x0k/json-schema-merge.
Step 4
Custom processors
User-defined functions: removeExtensions · removePaths · removeParameters · traverse.
Runtime URL schemas · validate()

Runs in the browser (or standalone) via lib/oas/parser/index.ts. Fetched on demand and reprocessed on every cold load.

Step 1
dereference()
Custom resolver. Replaces every $ref inline (sibling properties are lost — fast and total).
Step 2
upgradeSchema()
Custom 3.0 → 3.1. Always converts example → examples regardless of version.
Step 3
flattenAllOf()
Same merge semantics as the build pipeline, but a runtime-friendly implementation.
Step 4
Feed GraphQL
The normalized doc flows into the same Pothos schema as file pipeline output.
Shared GraphQL layer · lib/oas/graphql/index.ts

Both pipelines feed a Pothos schema served in-browser by GraphQL Yoga. The React components query it via a tiny GraphQLClient. This is what turns a normalized OpenAPI doc into the components you see on screen.

Custom scalar

JSONSchemaScalar

Serializes raw schemas through handleCircularRefs() so deeply nested or recursive types survive serialization.

Resolver

ExampleItem

Resolves media-type level example/examples into a consistent array shape before reaching the client.

09UI · icons · theming

A shadcn-style component kit in packages/zudoku/src/lib/ui/, Lucide icons re-exported through zudoku/icons, and a Tailwind v4 theme driven by CSS variables. Themes can be authored as shadcn registry items.

50+ Radix-based primitives: Accordion · Button · Callout · Card · CodeBlock · CodeTabs · Dialog · Drawer · DropdownMenu · NavigationMenu · Popover · Select · Sheet · Stepper · SyntaxHighlight · Tabs · Tooltip — and many more.

Icons

zudoku/icons

Wraps Lucide so users get tree-shakable, typed icons that match the rest of the system. A custom MissingIcon fills gaps.

Theme

vite/plugin-theme.ts

Generates virtual:zudoku-theme.css from the user's theme config. Reads shadcn registry items, emits CSS vars for light + dark, wired through next-themes.
